# Requirements Document

## Introduction

FlowPilot AI is an autonomous business workflow platform that functions as an AI Operations Manager. The platform ingests real business communications (emails, invoices, PDFs, contracts), classifies intent using an AI model, routes each item to one of four specialized LangGraph agents (Sales, Support, Finance, Executive), executes structured workflows, and surfaces results on a live dashboard. The system targets hackathon demonstration with a Supabase PostgreSQL backend that resets weekly, custom JWT-based authentication, a Next.js + FastAPI stack, and animated workflow visualizations powered by Framer Motion.

---

## Glossary

- **Platform**: The full FlowPilot AI application (frontend + backend combined).
- **Frontend**: The Next.js / React / TypeScript application served to the browser.
- **Backend**: The FastAPI Python application that exposes REST endpoints.
- **Auth Service**: The custom JWT-based authentication module built within the Backend.
- **AI Inbox**: The UI section where users submit email text or document uploads for processing.
- **Intent Detector**: The AI component (GPT-based) that classifies the intent of an incoming item and assigns a confidence score.
- **Agent Router**: The LangGraph orchestration layer that dispatches work to specialized agents.
- **Sales Agent**: The LangGraph agent responsible for sales inquiry handling.
- **Support Agent**: The LangGraph agent responsible for customer support handling.
- **Finance Agent**: The LangGraph agent responsible for invoice and payment handling.
- **Executive Agent**: The LangGraph agent that summarizes business activity and recommends actions.
- **Workflow**: A directed sequence of steps executed by an agent in response to a classified item.
- **Workflow Viewer**: The animated UI component that visualises a Workflow's step-by-step execution.
- **Dashboard**: The main UI page displaying live business metrics, recent activity, and AI suggestions.
- **OCR Engine**: The Tesseract-based component that extracts structured fields from uploaded document images/PDFs.
- **CRM Lead**: A structured record (name, company, contact, status) created by the Sales Agent and stored in Supabase.
- **Quotation**: A document generated by the Sales Agent containing pricing information in response to a sales inquiry.
- **Ticket**: A support record created by the Support Agent in Supabase.
- **Invoice Record**: A structured record extracted from an uploaded invoice document, stored in Supabase.
- **JWT**: JSON Web Token used for stateless authentication.
- **Supabase**: The hosted PostgreSQL database used for persistent storage (resets weekly in hackathon context).

## Requirements

### Requirement 1 — Custom Authentication

**User Story:** As a user, I want to register and log in with my own credentials so that only authorised users can access the platform.

#### Acceptance Criteria

1. THE Auth Service SHALL provide a registration endpoint that accepts an email address and a plaintext password, stores the email and a bcrypt-hashed password in Supabase, and returns a signed JWT on success.
2. THE Auth Service SHALL provide a login endpoint that accepts an email address and a plaintext password, verifies the password against the stored bcrypt hash, and returns a signed JWT on success.
3. IF the login endpoint receives an email that does not exist or a password that does not match the stored hash, THEN THE Auth Service SHALL return an HTTP 401 response with an error message.
4. THE Auth Service SHALL sign JWTs with an RS256 or HS256 algorithm using a secret key stored in a server-side environment variable.
5. WHEN a protected Backend endpoint receives a request, THE Auth Service SHALL validate the JWT from the Authorization header and reject requests with an invalid or expired token with HTTP 401.
6. THE Frontend SHALL store the JWT in memory or an httpOnly cookie and SHALL include the JWT in the Authorization header of every authenticated API request.
7. THE Frontend SHALL display a login page at the `/login` route that accepts email and password and redirects the authenticated user to the Dashboard on success.
8. IF a JWT expires, THEN THE Frontend SHALL redirect the user to the `/login` route and clear the stored token.

### Requirement 2 — AI Inbox

**User Story:** As a user, I want to submit real email text or upload a document so that the platform can automatically classify it and route it to the correct agent.

#### Acceptance Criteria

1. THE AI Inbox SHALL accept plain-text email input via a text area with a maximum length of 10,000 characters.
2. THE AI Inbox SHALL accept file uploads in PDF, PNG, JPG, and JPEG formats with a maximum file size of 10 MB.
3. WHEN the user submits an email text or file, THE Intent Detector SHALL classify the item into one of the following categories: `sales_inquiry`, `support_complaint`, `support_faq`, `invoice`, `contract`, or `general`.
4. WHEN the Intent Detector classifies an item, THE Intent Detector SHALL return a confidence score between 0.0 and 1.0 alongside the classification label.
5. THE AI Inbox SHALL display the classification label and confidence score to the user after classification.
6. IF the Intent Detector returns a confidence score below 0.5, THEN THE AI Inbox SHALL display a low-confidence warning and allow the user to manually select a category before routing.
7. WHEN an item is classified and confirmed, THE Agent Router SHALL dispatch the item to the appropriate agent based on the classification mapping: `sales_inquiry` → Sales Agent; `support_complaint` or `support_faq` → Support Agent; `invoice` → Finance Agent; `general` → Executive Agent.
8. THE AI Inbox SHALL display the Workflow Viewer animation from the moment of submission until the Workflow reaches a terminal state.

---

### Requirement 3 — Sales Agent

**User Story:** As a user, I want the Sales Agent to automatically respond to sales inquiries so that leads are captured and replies are prepared without manual effort.

#### Acceptance Criteria

1. WHEN the Sales Agent receives a `sales_inquiry` item, THE Sales Agent SHALL generate a Quotation containing at minimum: item descriptions inferred from the inquiry, unit prices, and a total amount.
2. WHEN the Sales Agent receives a `sales_inquiry` item, THE Sales Agent SHALL create a CRM Lead record in Supabase containing: sender name (extracted or inferred), company (extracted or inferred), contact email, inquiry summary, and status set to `new`.
3. WHEN the Sales Agent receives a `sales_inquiry` item, THE Sales Agent SHALL draft a reply email addressed to the sender that includes the Quotation content.
4. WHEN the Sales Agent receives a `sales_inquiry` item, THE Sales Agent SHALL generate a meeting suggestion containing a proposed agenda and a suggested time slot.
5. THE Sales Agent SHALL complete all four actions (Quotation, CRM Lead, reply email, meeting suggestion) within 30 seconds of receiving the item.
6. THE Sales Agent SHALL store the Quotation and drafted reply in Supabase linked to the CRM Lead record.

---

### Requirement 4 — Support Agent

**User Story:** As a user, I want the Support Agent to handle customer complaints and FAQs so that support tickets are created and escalations are flagged automatically.

#### Acceptance Criteria

1. WHEN the Support Agent receives a `support_complaint` or `support_faq` item, THE Support Agent SHALL create a Ticket in Supabase containing: sender information, a summary of the issue, category (`complaint` or `faq`), and status set to `open`.
2. WHEN the Support Agent receives a `support_complaint` item, THE Support Agent SHALL generate a ticket summary of no more than 150 words.
3. WHEN the Support Agent receives a `support_faq` item, THE Support Agent SHALL generate a draft FAQ answer using the GPT model.
4. IF the Support Agent determines that a complaint has high severity (e.g., contains keywords such as "legal", "lawsuit", "urgent", "fraud"), THEN THE Support Agent SHALL set the Ticket status to `escalated` and flag it in the response payload.
5. THE Support Agent SHALL complete Ticket creation and summary generation within 20 seconds of receiving the item.

---

### Requirement 5 — Finance Agent

**User Story:** As a user, I want the Finance Agent to extract structured data from invoices so that payment details are available without manual data entry.

#### Acceptance Criteria

1. WHEN the Finance Agent receives an `invoice` item that is an image or PDF, THE OCR Engine SHALL extract text from the document before passing it to the Finance Agent.
2. WHEN the Finance Agent processes an invoice, THE Finance Agent SHALL extract and store the following fields in an Invoice Record in Supabase: vendor name, invoice number (if present), invoice date (if present), line items (description and amount), subtotal, GST/tax amount, total amount due, and payment due date.
3. IF a required field (vendor name or total amount due) cannot be extracted, THEN THE Finance Agent SHALL mark that field as `null` in the Invoice Record and include a partial extraction warning in the response.
4. THE Finance Agent SHALL complete OCR extraction and Invoice Record creation within 30 seconds of receiving the item.
5. THE Finance Agent SHALL return the extracted Invoice Record fields to the Frontend for display in the AI Inbox result panel.

---

### Requirement 6 — Executive Agent

**User Story:** As a user, I want the Executive Agent to summarise all business activity and suggest next actions so that I can quickly understand the state of operations.

#### Acceptance Criteria

1. WHEN the user sends a natural-language query to the Executive Agent (e.g., "What happened today?"), THE Executive Agent SHALL query Supabase for all Workflow results, CRM Leads, Tickets, and Invoice Records created within the current calendar day (UTC).
2. WHEN the Executive Agent has retrieved the day's data, THE Executive Agent SHALL generate a plain-language summary of no more than 300 words covering: total items processed, breakdown by category, notable CRM Leads, escalated Tickets, and Invoice totals.
3. THE Executive Agent SHALL generate a list of at least three recommended next actions based on the day's data.
4. THE Executive Agent SHALL complete the summary and recommendation generation within 20 seconds of receiving the query.

---

### Requirement 7 — Workflow Viewer

**User Story:** As a user, I want to see an animated visualisation of the workflow execution so that I understand what the AI is doing at each step.

#### Acceptance Criteria

1. THE Workflow Viewer SHALL render a directed graph of nodes representing the following fixed steps in order: `Email Received` → `Intent Detection` → `Agent Routing` → `Agent Execution` → `Actions` → `CRM / Reply / Record` → `Completed`.
2. WHEN a Workflow step becomes active, THE Workflow Viewer SHALL animate the corresponding node with a visual highlight (e.g., pulsing border or colour change) using Framer Motion.
3. WHEN a Workflow step completes successfully, THE Workflow Viewer SHALL mark the corresponding node with a success indicator and advance the highlight to the next step.
4. IF a Workflow step fails, THEN THE Workflow Viewer SHALL mark the corresponding node with an error indicator and halt the animation.
5. THE Workflow Viewer SHALL display the elapsed time in seconds from submission to completion below the graph.

---

### Requirement 8 — Dashboard

**User Story:** As a user, I want to view a live dashboard with business metrics and AI suggestions so that I have an operational overview of the platform.

#### Acceptance Criteria

1. THE Dashboard SHALL display the following live metrics fetched from the Backend on page load and refreshed every 30 seconds: Today's Requests (count), Completed Workflows (count), Pending Workflows (count), Total Revenue from Quotations (sum), and Open Support Tickets (count).
2. THE Dashboard SHALL render a workflow activity chart using Recharts showing the count of workflows by category (`sales_inquiry`, `support`, `invoice`, `general`) for the current day.
3. THE Dashboard SHALL display a Recent Activity feed listing the last 10 processed items with: timestamp, classification label, assigned agent, and status.
4. THE Dashboard SHALL display an AI Suggestions panel populated by the Executive Agent with at least three recommended next actions, refreshed each time the Executive Agent is queried.
5. WHEN no data exists for the current day (e.g., after a database reset), THE Dashboard SHALL display zero values for all metrics and an empty Recent Activity feed without errors.

---

### Requirement 9 — Document Intelligence

**User Story:** As a user, I want to upload invoices, PDFs, and contracts so that the platform extracts structured information automatically.

#### Acceptance Criteria

1. THE Frontend SHALL provide a Document Intelligence upload interface that accepts PDF, PNG, JPG, and JPEG files up to 10 MB.
2. WHEN a document is uploaded, THE Backend SHALL pass the file to the OCR Engine (Tesseract) and return the raw extracted text to the calling agent.
3. THE Document Intelligence page SHALL display the extracted fields from the Invoice Record after Finance Agent processing, grouped as: Vendor Information, Line Items, Tax & Totals, and Payment Details.
4. IF the uploaded file format is not PDF, PNG, JPG, or JPEG, THEN THE Backend SHALL return an HTTP 400 response with a descriptive error message.
5. THE OCR Engine SHALL process uploaded documents using Tesseract with language set to English (`eng`) as the default.

---

### Requirement 10 — Analytics

**User Story:** As a user, I want to view analytics charts so that I can monitor agent performance and workflow trends over time.

#### Acceptance Criteria

1. THE Analytics page SHALL display the following charts using Recharts: Requests Today (line or bar chart by hour), Sales Inquiries vs Support Tickets (grouped bar chart by day for the last 7 days), Pending Tasks (pie chart by agent), and Agent Usage (bar chart showing workflow count per agent).
2. WHEN the Analytics page loads, THE Frontend SHALL fetch aggregated metrics from a dedicated Backend analytics endpoint.
3. THE Analytics page SHALL refresh chart data every 60 seconds without requiring a full page reload.
4. IF no data exists for a given time range, THEN THE Analytics page SHALL render empty chart states with a "No data available" label rather than an error.

---

### Requirement 11 — LangGraph Multi-Agent Architecture

**User Story:** As a platform operator, I want all agents to be implemented as LangGraph graphs so that the multi-agent orchestration is structured, inspectable, and extensible.

#### Acceptance Criteria

1. THE Agent Router SHALL be implemented as a LangGraph StateGraph that routes each classified item to the correct agent node based on the intent label.
2. EACH of the four agents (Sales, Support, Finance, Executive) SHALL be implemented as a distinct LangGraph StateGraph with defined input state, output state, and at least two internal nodes representing discrete workflow steps.
3. THE Backend SHALL expose a `/api/process` POST endpoint that accepts an item payload (text or file reference, classification label, confidence score) and returns the final agent output and a list of completed workflow steps.
4. THE Backend SHALL expose a `/api/agents/status` GET endpoint that returns the current status of all active LangGraph workflow runs.
5. THE Agent Router SHALL pass the full item context (raw text, extracted OCR text if applicable, metadata) to the dispatched agent as part of the LangGraph state.

---

### Requirement 12 — Data Persistence and Reset Resilience

**User Story:** As a platform operator, I want the system to handle Supabase weekly resets gracefully so that the platform remains functional after a database reset.

#### Acceptance Criteria

1. THE Backend SHALL define all Supabase table schemas using migration scripts or an initialisation endpoint so that tables can be recreated after a reset.
2. THE Backend SHALL expose a `/api/admin/init-db` POST endpoint (protected by JWT) that creates all required tables if they do not exist.
3. WHEN the Dashboard or Analytics page detects that a Supabase query returns a "relation does not exist" error, THE Frontend SHALL display a "Database not initialised" banner with a button to trigger the `/api/admin/init-db` endpoint.
4. THE Platform SHALL not store any state solely in memory that cannot be reconstructed from Supabase after a restart.
